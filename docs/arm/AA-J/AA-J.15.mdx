<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} Pragmas can be used as an alternative to aspect_specifications to specify certain aspects. </p>

#### Name Resolution Rules

<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} Certain <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s are defined to be <em>program unit pragmas</em>. A <code><a href="../AA-4/AA-4.1#S0091">name</a></code> given as the argument of a program unit pragma shall resolve to denote the declarations or renamings of one or more program units that occur immediately within the declarative region or <code><a href="../AA-10/AA-10.1#S0285">compilation</a></code> in which the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> immediately occurs, or it shall resolve to denote the declaration of the immediately enclosing program unit (if any); the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> applies to the denoted program unit(s). If there are no <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s given as arguments, the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> applies to the immediately enclosing program unit. </p>
<Admonition type="note">
<p><strong>Ramification: </strong>The fact that this is a Name Resolution Rule means that the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> will not apply to declarations from outer declarative regions. </p></Admonition>


#### Legality Rules

<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} A program unit pragma shall appear in one of these places: </p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} At the place of a <code><a href="../AA-10/AA-10.1#S0286">compilation_unit</a></code>, in which case the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> shall immediately follow in the same <code><a href="../AA-10/AA-10.1#S0285">compilation</a></code> (except for other <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s) a <code><a href="../AA-10/AA-10.1#S0288">library_unit_declaration</a></code> that is a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>, <code><a href="../AA-12/AA-12.1#S0311">generic_subprogram_declaration</a></code>, or <code><a href="../AA-12/AA-12.3#S0315">generic_instantiation</a></code>, and the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> shall have an argument that is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> denoting that declaration. </p>
<Admonition type="note">
<p><strong>Ramification: </strong>The <code><a href="../AA-4/AA-4.1#S0091">name</a></code> has to denote the immediately preceding <code><a href="../AA-10/AA-10.1#S0288">library_unit_declaration</a></code>. </p></Admonition>

<p>{"{"}<em>8652/0033</em>{"}"} {"{"}<em>{"AI95-00136-01"}</em>{"}"} {"{"}<em>{"AI12-0417-1"}</em>{"}"} Immediately within the visible part of a program unit and before any nested declaration (but not within a generic formal part), in which case the argument, if any, shall be a <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> that denotes the immediately enclosing program unit declaration. </p>
<Admonition type="note">
<p><strong>Ramification: </strong>The argument is optional in this case. </p></Admonition>

<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} At the place of a declaration other than the first, of a <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> or program unit declaration, in which case the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> shall have an argument, which shall be a <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> that denotes one or more of the following (and nothing else): a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>, a <code><a href="../AA-12/AA-12.1#S0311">generic_subprogram_declaration</a></code>, or a <code><a href="../AA-12/AA-12.3#S0315">generic_instantiation</a></code>, of the same <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code> or program unit declaration. </p>
<Admonition type="note">
<p><strong>Ramification: </strong>If you want to denote a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> that is not a completion, or a <code><a href="../AA-7/AA-7.1#S0229">package_declaration</a></code>, for example, you have to put the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> inside. </p></Admonition>

<p>{"{"}<em>{"AI05-0132-1"}</em>{"}"} {"{"}<em>{"AI12-0417-1"}</em>{"}"} Certain program unit pragmas are defined to be <em>library unit pragmas</em>. If a library unit pragma applies to a program unit, the program unit shall be a library unit. </p>
<Admonition type="note">
<p><strong>Ramification: </strong>This, together with the rules for program unit pragmas above, implies that if a library unit pragma applies to a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> (and similar things), it has to appear immediately after the <code><a href="../AA-10/AA-10.1#S0286">compilation_unit</a></code>, whereas if the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> applies to a <code><a href="../AA-7/AA-7.1#S0229">package_declaration</a></code>, a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> that is not a completion (and similar things), it has to appear inside, as the first <code><a href="../AA-3/AA-3.11#S0087">declarative_item</a></code>. </p></Admonition>


#### Static Semantics

<p>{"{"}<em>8652/0034</em>{"}"} {"{"}<em>{"AI95-00041-01"}</em>{"}"} {"{"}<em>{"AI12-0417-1"}</em>{"}"} A library unit pragma that applies to a generic unit does not apply to its instances, unless a specific rule for the pragma specifies the contrary. </p>

#### Implementation Advice

<p>{"{"}<em>8652/0034</em>{"}"} {"{"}<em>{"AI95-00041-01"}</em>{"}"} {"{"}<em>{"AI12-0417-1"}</em>{"}"} When applied to a generic unit, a program unit pragma that is not a library unit pragma should apply to each instance of the generic unit for which there is not an overriding pragma applied directly to the instance. </p>
<Admonition type="note">
<p><strong>Implementation Advice: </strong>When applied to a generic unit, a program unit pragma that is not a library unit pragma should apply to each instance of the generic unit for which there is not an overriding pragma applied directly to the instance.</p></Admonition>


#### Wording Changes from Ada 2005

<Admonition type="note">
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} {"{"}<em>{"AI05-0299-1"}</em>{"}"} This subclause is new. Many existing pragmas have been converted into aspects; the pragmas have moved here.</p></Admonition>

<Admonition type="note">
<p>{"{"}<em>{"AI05-0132-1"}</em>{"}"} {"{"}<em>{"AI12-0417-1"}</em>{"}"} <strong>Correction:</strong> A library unit pragma must apply directly to a library unit, even if no name is given in the pragma. </p></Admonition>


#### Wording Changes from Ada 2012

<Admonition type="note">
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} The terms "program unit pragma" and "library unit pragma" were moved here as all of the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s that use these terms are now in this annex. </p></Admonition>


## J.15.1  Pragma Inline


#### Syntax

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The form of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Inline, which is a program unit pragma (see 10.1.5), is as follows: </p>
<p>  <strong>pragma</strong> Inline (<code><a href="../AA-4/AA-4.1#S0091">name</a></code>{"{"}, <code><a href="../AA-4/AA-4.1#S0091">name</a></code>{"}"}); </p>

#### Legality Rules

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> shall apply to one or more callable entities or generic subprograms. </p>

#### Static Semantics

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} <code><a href="../AA-2/AA-2.8#S0019">Pragma</a></code> Inline specifies that the Inline aspect (see 6.3.2) for each entity denoted by each <code><a href="../AA-4/AA-4.1#S0091">name</a></code> given in the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> has the value True.</p>
<Admonition type="note">
<p><strong>Ramification: </strong>Note that inline expansion is desired no matter what <code><a href="../AA-4/AA-4.1#S0091">name</a></code> is used in the call. This allows one to request inlining for only one of several overloaded subprograms as follows:</p></Admonition>

<CodeBlock>
package IO is{"\n"}   procedure Put(X : in Integer);{"\n"}   procedure Put(X : in String);{"\n"}   procedure Put(X : in Character);{"\n"}private{"\n"}   procedure Character_Put(X : in Character) renames Put;{"\n"}   pragma Inline(Character_Put);{"\n"}end IO;{"\n"}
</CodeBlock>

<CodeBlock>
with IO; use IO;{"\n"}procedure Main is{"\n"}   I : Integer;{"\n"}   C : Character;{"\n"}begin{"\n"}   ...{"\n"}   Put(C); -- Inline expansion is desired.{"\n"}   Put(I); -- Inline expansion is NOT desired.{"\n"}end Main;{"\n"}
</CodeBlock>


#### Implementation Permissions

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} An implementation may allow a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Inline that has an argument which is a <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> denoting a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> of the same <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>.</p>
<Admonition type="note">
<p><strong>Reason: </strong>This is allowed for Ada 83 compatibility. This is only a permission as this usage was considered obsolescent even for Ada 95. </p></Admonition>

<Admonition type="note">
<p><strong>Discussion: </strong>We only need to allow this in <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>s, because a <code><a href="../AA-3/AA-3.11#S0089">body</a></code> is only allowed in another <code><a href="../AA-3/AA-3.11#S0089">body</a></code>, and these all have <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>s. </p></Admonition>

<Admonition type="note">
<p>NOTE 1   {"{"}<em>{"AI05-0229-1"}</em>{"}"} {"{"}<em>{"AI12-0440-1"}</em>{"}"} The name in a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Inline can denote more than one entity in the case of overloading. Such a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> applies to all of the denoted entities. </p></Admonition>


#### Incompatibilities With Ada 83

<Admonition type="note">
<p>{"{"}<em>{"AI95-00309-01"}</em>{"}"} {"{"}<em>{"AI05-0229-1"}</em>{"}"} A pragma Inline cannot refer to a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> outside of that body. The pragma can be given inside of the subprogram body. Ada 2005 adds an Implementation Permission to allow this usage for compatibility (and Ada 95 implementations also can use this permission), but implementations do not have to allow such <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s. </p></Admonition>


#### Extensions to Ada 83

<Admonition type="note">
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Inline is allowed inside a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> if there is no corresponding <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>. This is for uniformity with other program unit pragmas. </p></Admonition>


#### Extensions to Ada 95

<Admonition type="note">
<p>{"{"}<em>{"AI95-00309-01"}</em>{"}"} {"{"}<em>{"AI05-0229-1"}</em>{"}"} <strong>Amendment Correction:</strong> Implementations are allowed to let <code><a href="../AA-2/AA-2.8#S0019">Pragma</a></code> Inline apply to a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code>. </p></Admonition>


#### Wording Changes from Ada 2005

<Admonition type="note">
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} {"{"}<em>{"AI05-0299-1"}</em>{"}"} This subclause is new. Pragma Inline was moved here from 6.3.2; aspect Inline lives there now. </p></Admonition>


## J.15.2  Pragma No_Return


#### Syntax

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The form of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> No_Return, which is a representation pragma (see 13.1), is as follows: </p>
<p>{"{"}<em>{"AI12-0269-1"}</em>{"}"}   <strong>pragma</strong> No_Return (<em>subprogram_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>{"{"}, <em>subprogram_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>{"}"}); </p>

#### Legality Rules

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} {"{"}<em>{"AI12-0269-1"}</em>{"}"} Each <em>subprogram_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> shall denote one or more subprograms or generic subprograms. [The <em>subprogram_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> shall not denote a null procedure nor an instance of a generic unit.] </p>

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} {"{"}<em>{"AI12-0269-1"}</em>{"}"} <code><a href="../AA-2/AA-2.8#S0019">Pragma</a></code> No_Return specifies that the No_Return aspect (see 6.5.1) for each subprogram denoted by each <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> given in the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> has the value True. </p>

## J.15.3  Pragma Pack


#### Syntax

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The form of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Pack, which is a representation pragma (see 13.1), is as follows: </p>
<p>  <strong>pragma</strong> Pack (<em>first_subtype_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>); </p>

#### Legality Rules

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The <em>first_subtype_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Pack shall denote a composite subtype. </p>

#### Static Semantics

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} <code><a href="../AA-2/AA-2.8#S0019">Pragma</a></code> Pack specifies that the Pack aspect (see 13.2) for the type denoted by <em>first_subtype_</em><code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> has the value True. </p>

#### Wording Changes from Ada 2005

<Admonition type="note">

## J.15.4  Pragma Storage_Size


#### Syntax

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The form of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Storage_Size is as follows: </p>
<p>  <strong>pragma</strong> Storage_Size (<code><a href="../AA-4/AA-4.4#S0132">expression</a></code>);</p>
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Storage_Size is allowed only immediately within a <code><a href="../AA-9/AA-9.1#S0246">task_definition</a></code>. </p>

#### Name Resolution Rules

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Storage_Size is expected to be of any integer type. </p>

#### Static Semantics

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Storage_Size sets the Storage_Size aspect (see 13.3) of the type defined by the immediately enclosing <code><a href="../AA-9/AA-9.1#S0246">task_definition</a></code> to the value of the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>. </p>

#### Wording Changes from Ada 2005

<Admonition type="note">
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} {"{"}<em>{"AI05-0299-1"}</em>{"}"} This subclause is new. Pragma Storage_Size was moved here from 13.3; aspect Storage_Size lives there now. </p></Admonition>


## J.15.5  Interfacing Pragmas


#### Syntax

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} An <em>interfacing pragma</em> is a representation pragma that is one of the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s Import, Export, or Convention. Their forms are as follows: </p>
<p>  <strong>pragma</strong> Import(     [Convention ={">"}] <em>convention_</em><code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>, [Entity ={">"}] <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>  [, [External_Name ={">"}] <em>external_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>]  [, [Link_Name ={">"}] <em>link_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>]);</p>
<p>  <strong>pragma</strong> Export(     [Convention ={">"}] <em>convention_</em><code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>, [Entity ={">"}] <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>  [, [External_Name ={">"}] <em>external_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>]  [, [Link_Name ={">"}] <em>link_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>]);</p>
<p>  <strong>pragma</strong> Convention([Convention ={">"}] <em>convention_</em><code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>,[Entity ={">"}] <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code>);</p>
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} For <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s Import and Export, the argument for Link_Name shall not be given without the <em>pragma_argument_</em><code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> unless the argument for External_Name is given. </p>

#### Name Resolution Rules

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"}  The expected type for an <em>external_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> and a <em>link_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> in an interfacing pragma is String. </p>

#### Legality Rules

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The <em>convention_</em><code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> of an interfacing pragma shall be the name of a convention (see B.1).</p>
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import shall be the completion of a declaration. Notwithstanding any rule to the contrary, a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import may serve as the completion of any kind of (explicit) declaration if supported by an implementation for that kind of declaration. If a completion is a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import, then it shall appear in the same <code><a href="../AA-3/AA-3.11#S0086">declarative_part</a></code>, <code><a href="../AA-7/AA-7.1#S0230">package_specification</a></code>, <code><a href="../AA-9/AA-9.1#S0246">task_definition</a></code>, or <code><a href="../AA-9/AA-9.4#S0251">protected_definition</a></code> as the declaration. For a library unit, it shall appear in the same <code><a href="../AA-10/AA-10.1#S0285">compilation</a></code>, before any subsequent <code><a href="../AA-10/AA-10.1#S0286">compilation_unit</a></code>s other than <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s. If the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> denotes more than one entity, then the <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import is the completion of all of them.</p>
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The <em>external_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> and <em>link_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code> of a <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import or Export shall be static.</p>
<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} The <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> of each of these pragmas shall denote a declaration that may have the similarly named aspect specified. </p>

#### Static Semantics

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} An interfacing pragma specifies various aspects of the entity denoted by the <code><a href="../AA-13/AA-13.1#S0345">local_name</a></code> as follows: </p>
<p>The Convention aspect (see B.1) is <em>convention_</em><code><a href="../AA-2/AA-2.3#S0002">identifier</a></code>.</p>
<p>A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import specifies that the Import aspect (see B.1) is True.</p>
<p>A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Export specifies that the Export aspect (see B.1) is True.</p>
<p>For both <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Import and Export, if an external name is given in the pragma, the External_Name aspect (see B.1) is specified to be <em>external_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. If a link name is given in the pragma, the Link_Name aspect (see B.1) is specified to be the <em>link_name_string_</em><code><a href="../AA-4/AA-4.4#S0132">expression</a></code>. </p>

#### Wording Changes from Ada 2005

## J.15.7  Pragmas Interrupt_Handler and Attach_Handler


#### Syntax

In addition to the places where Legality Rules normally apply (see 12.3), these rules also apply in the private part of an instance of a generic unit. </p>
<Admonition type="note">
<p><strong>Discussion: </strong>In the case of a <code><a href="../AA-9/AA-9.4#S0249">protected_type_declaration</a></code>, an <code><a href="../AA-3/AA-3.3#S0032">object_declaration</a></code> of an object of that type need not be at library level.</p></Admonition>

<Admonition type="note">
<p>{"{"}<em>{"AI05-0033-1"}</em>{"}"} We cannot allow these pragmas in a generic body, because legality rules are not checked for instance bodies, and these should not be allowed if the instance is not at the library level. The protected types can be declared in the private part if this is desired. Note that while the 'Access to use the handler would provide the check in the case of Interrupt_Handler, there is no other check for Attach_Handler. Since these pragmas are so similar, we want the rules to be the same. </p></Admonition>


#### Static Semantics

<p>{"{"}<em>{"AI05-0229-1"}</em>{"}"} For an implementation that supports Annex C, a pragma Interrupt_Handler specifies the Interrupt_Handler aspect (see C.3.1) for the protected procedure <em>handler_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> to have the value True. For an implementation that supports Annex C, a pragma Attach_Handler specifies the Attach_Handler aspect (see C.3.1) for the protected procedure <em>handler_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> to have the value of the given <code><a href="../AA-4/AA-4.4#S0132">expression</a></code>[ as evaluated at object creation time]. </p>

<p>  <strong>pragma</strong> Preelaborable_Initialization(<code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code>);</p>
<p>  <strong>pragma</strong> Pure[(<em>library_unit_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>)];</p>
<p>  <strong>pragma</strong> Elaborate_Body[(<em>library_unit_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>)];</p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} <code><a href="../AA-2/AA-2.8#S0019">Pragma</a></code>s Preelaborate, Pure, and Elaborate_Body are library unit pragmas. </p>
<Admonition type="note">
<p><strong>Ramification: </strong><code><a href="../AA-2/AA-2.8#S0019">Pragma</a></code> Preelaborable_Initialization is <strong>not</strong> a library unit pragma. </p></Admonition>


#### Static Semantics

<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Preelaborate specifies that a library unit is preelaborated, namely that the Preelaborate aspect (see 10.2.1) of the library unit is True.</p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Pure specifies that a library unit is declared pure, namely that the Pure aspect (see 10.2.1) of the library unit is True.</p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Elaborate_Body specifies that [a library unit requires a completion, namely that] the Elaborate_Body aspect (see 10.2.1) of the library unit is True. </p>

#### Legality Rules

<p>{"{"}<em>{"AI12-0409-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Preelaborable_Initialization specifies that the Preelaborable_Initialization aspect (see 10.2.1) for a composite type is True. This pragma shall appear in the visible part of a package or generic package.</p>
<p>{"{"}<em>{"AI12-0409-1"}</em>{"}"} If the pragma appears in the first declaration list of a <code><a href="../AA-7/AA-7.1#S0230">package_specification</a></code>, then the <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> shall denote the first subtype of a composite type, and the type shall be declared immediately within the same package as the pragma. The composite type shall be one for which the Preelaborable_Initialization aspect can be directly specified as True. In addition to the places where Legality Rules normally apply (see 12.3), these rules also apply in the private part of an instance of a generic unit.</p>
<p>{"{"}<em>{"AI12-0409-1"}</em>{"}"} If the pragma appears in a <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code>, then the <code><a href="../AA-4/AA-4.1#S0092">direct_name</a></code> shall denote a type declared in the same <code><a href="../AA-12/AA-12.1#S0313">generic_formal_part</a></code> as the pragma, and be one for which the Preelaborable_Initialization aspect can be directly specified as True. </p>
<Admonition type="note">
<p>NOTE 1   {"{"}<em>{"AI12-0409-1"}</em>{"}"} Pragmas Elaborate and Elaborate_All, which do not have associated aspects, are found in 10.2.1. </p></Admonition>

## J.15.15  Distribution Pragmas

<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} This subclause defines pragmas that specify properties of units for distributed systems.</p>
<Admonition type="note">
<p><strong>Discussion: </strong>{"{"}<em>{"AI12-0417-1"}</em>{"}"} These pragmas are presented in the order that the aspects are defined in Annex E. </p></Admonition>


#### Syntax

<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} The following <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code>s are defined with the given forms: </p>
<p>  <strong>pragma</strong> Shared_Passive[(<em>library_unit_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>)];</p>
<p>  <strong>pragma</strong> Remote_Types[(<em>library_unit_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>)];</p>
<p>  <strong>pragma</strong> Remote_Call_Interface[(<em>library_unit_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>)];</p>
<p>  <strong>pragma</strong> All_Calls_Remote[(<em>library_unit_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>)];</p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} Each of these pragmas is a library unit pragma. </p>

#### Static Semantics

<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} A <em>categorization pragma</em> is a pragma that specifies a corresponding categorization aspect.</p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} The pragmas Shared_Passive, Remote_Types, and Remote_Call_Interface are categorization pragmas. In addition, the pragma Pure (see J.15.14) is considered a categorization pragma. </p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Shared_Passive specifies that a library unit is a shared passive library unit, namely that the Shared_Passive aspect (see E.2.1) of the library unit is True.</p>
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Remote_Types specifies that a library unit is a remote types library unit, namely that the Remote_Types aspect (see E.2.2) of the library unit is True.</p>
<p>A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> Remote_Call_Interface specifies that a library unit is a remote call interface, namely that the Remote_Call_Interface aspect (see E.2.3) of the library unit is True.</p>
<p>A <code><a href="../AA-2/AA-2.8#S0019">pragma</a></code> All_Calls_Remote specifies that the All_Calls_Remote aspect (see E.2.3) of the library unit is True.</p>

#### Wording Changes from Ada 2012

<Admonition type="note">
<p>{"{"}<em>{"AI12-0417-1"}</em>{"}"} This subclause is new. These pragmas were moved here from Annex E, "Distributed Systems". </p></Admonition>
