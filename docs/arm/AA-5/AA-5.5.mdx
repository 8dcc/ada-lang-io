#### Name Resolution Rules

<p>{"{"}<em>{"AI12-0251-1"}</em>{"}"} <br />In a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> that is an <em>integer_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code>, the <em>integer_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code> is expected to be of any integer type. <br /></p>


#### Static Semantics

<p>{"{"}<em>{"AI12-0061-1"}</em>{"}"} <br />A <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> declares a <em>loop parameter</em>, which is an object whose subtype (and nominal subtype) is that defined by the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>. <br /></p>

<p>{"{"}<em>{"AI12-0251-1"}</em>{"}"} <br />In a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> that has a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>, the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> declares a <em>chunk parameter</em> object whose subtype (and nominal subtype) is that defined by the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>. <br /></p>


#### Dynamic Semantics

<p>{"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0327-1"}</em>{"}"} <br />The <em>filter</em> of an <em>iterator construct</em>  (a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>, <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>, or <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code>) is defined to be <em>satisfied</em> when there is no <code><a href="../AA-5/AA-5.5#S0182">iterator_filter</a></code> for the iterator construct, or when the <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> of the <code><a href="../AA-5/AA-5.5#S0182">iterator_filter</a></code> evaluates to True for a given iteration of the iterator construct.<br /></p>

<Admonition type="aarm" aarm="glossary-entry" title="Glossary entry: ">
<strong></strong>An iterator filter is a construct that is used to restrict the elements produced by an iteration to those for which a boolean condition evaluates to True.<br /></Admonition>

<p>{"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0327-1"}</em>{"}"} <br />If a <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with an iterator construct is said to be <em>conditionally executed</em>, then the <code><a href="../AA-5/AA-5.1#S0167">statement</a></code>s are executed only when the filter of the iterator construct is satisfied.<br /></p>

<p>{"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0327-1"}</em>{"}"} <br />The loop iterators <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> and <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code> can also be used in contexts other than <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>s (for example, see 4.3.5 and 4.5.8). In such a context, the iterator <em>conditionally produces</em> values in the order specified for the associated construct below or in 5.5.2. The values produced are the values given to the loop parameter when the filter of the iterator construct is satisfied for that value. [No value is produced when the <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> of an <code><a href="../AA-5/AA-5.5#S0182">iterator_filter</a></code> evaluates to False.]<br /></p>

<p>{"{"}<em>{"AI12-0119-1"}</em>{"}"} <br />For the execution of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>, the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is executed zero or more times, until the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. The <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> is complete when a transfer of control occurs that transfers control out of the loop, or, in the case of an <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code>, as specified below.<br /></p>

<p>For the execution of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with a <strong>while</strong> <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code>, the condition is evaluated before each execution of the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code>; if the value of the <code><a href="../AA-4/AA-4.5#S0150">condition</a></code> is True, the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is executed; if False, the execution of the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> is complete.<br /></p>

<p>{"{"}<em>{"AI12-0251-1"}</em>{"}"} {"{"}<em>{"AI12-0294-1"}</em>{"}"} <br />If the reserved word <strong>parallel</strong> is present in the <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> (a <em>parallel loop</em>), the iterations are partitioned into one or more <em>chunks</em>, each with its own separate logical thread of control (see clause 9). If a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> is present in a parallel loop, it is elaborated first, and the result of the elaboration determines the maximum number of chunks used for the parallel loop. If the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> is an <em>integer_</em><code><a href="../AA-4/AA-4.4#S0138">simple_expression</a></code>, the elaboration evaluates the expression, and the value of the expression determines the maximum number of chunks. If a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> is present, the elaboration elaborates the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>, which defines the subtype of the chunk parameter, and the number of values in this subtype determines the maximum number of chunks. After elaborating the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>, a check is made that the determined maximum number of chunks is greater than zero. If this check fails, Program_Error is raised. <br /></p>

<p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} {"{"}<em>{"AI05-0262-1"}</em>{"}"} {"{"}<em>{"AI12-0071-1"}</em>{"}"} {"{"}<em>{"AI12-0119-1"}</em>{"}"} {"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0251-1"}</em>{"}"} {"{"}<em>{"AI12-0294-1"}</em>{"}"} {"{"}<em>{"AI12-0355-2"}</em>{"}"} {"{"}<em>{"AI12-0416-1"}</em>{"}"} <br />For the execution of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> that has an <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> including a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code>, after elaborating the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> and <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>, if any, the <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> is elaborated. This elaborates the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>, which defines the subtype of the loop parameter. If the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> defines a subtype with a null range, the execution of the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. Otherwise, the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is conditionally executed once for each value of the discrete subtype defined by the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> that satisfies the predicates of the subtype (or until the loop is left as a consequence of a transfer of control). Prior to each such iteration, the corresponding value of the discrete subtype is assigned to the loop parameter associated with the given iteration. If the loop is a parallel loop, each chunk has its own logical thread of control with its own copy of the loop parameter; otherwise (a <em>sequential loop</em>), a single logical thread of control performs the loop, and there is a single copy of the loop parameter. Each logical thread of control handles a distinct subrange of the values of the subtype of the loop parameter such that all values are covered with no overlaps. Within each logical thread of control, the values are assigned to the loop parameter in increasing order unless the reserved word <strong>reverse</strong> is present, in which case the values are assigned in decreasing order. In the absence of a transfer of control, the associated parallel construct of a <code><a href="../AA-5/AA-5.5#S0181">loop_parameter_specification</a></code> is complete when all of its logical threads of control are complete. <br /></p>

<p>{"{"}<em>{"AI12-0251-1"}</em>{"}"} {"{"}<em>{"AI12-0294-1"}</em>{"}"} <br />If a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> with a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> is present, then the logical thread of control associated with a given chunk has its own copy of the chunk parameter initialized with a distinct value from the discrete subtype defined by the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>. The values of the chunk parameters are assigned such that they increase with increasing values of the ranges covered by the corresponding loop parameters.<br /></p>

#### Examples

<p><em>Example of a loop statement without an iteration scheme:</em> <br /></p>

<CodeBlock>
loop{"\n"}   Get(Current_Character);{"\n"}   exit when Current_Character = '*';{"\n"}end loop;{"\n"}

</CodeBlock>
<p><em>Example of a loop statement with a <strong>while</strong> iteration scheme:</em> <br /></p>

<CodeBlock>
while Bid(N).Price {"<"} Cut_Off.Price loop{"\n"}   Record_Bid(Bid(N).Price);{"\n"}   N := N + 1;{"\n"}end loop;{"\n"}

</CodeBlock>
<p><em>Example of a loop statement with a <strong>for</strong> iteration scheme:</em> <br /></p>

<CodeBlock>
for J in Buffer'Range loop     --  works even with a null range{"\n"}   if Buffer(J) /= Space then{"\n"}      Put(Buffer(J));{"\n"}   end if;{"\n"}end loop;{"\n"}

</CodeBlock>
<p><em>Example of a loop statement with a name:</em> <br /></p>

<CodeBlock>
Summation:{"\n"}   while Next /= Head loop       -- see 3.10.1{"\n"}      Sum  := Sum + Next.Value;{"\n"}      Next := Next.Succ;{"\n"}   end loop Summation;{"\n"}

</CodeBlock>

#### Static Semantics

<p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} {"{"}<em>{"AI05-0269-1"}</em>{"}"} {"{"}<em>{"AI05-0292-1"}</em>{"}"} {"{"}<em>{"AI12-0156-1"}</em>{"}"} <br />An <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code> declares a <em>loop parameter</em>. <p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} {"{"}<em>{"AI05-0292-1"}</em>{"}"} <br />In a generalized iterator, the loop parameter is a constant. In an array component iterator, the loop parameter is a constant if the <em>iterable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> denotes a constant; otherwise it denotes a variable. In a container element iterator, the loop parameter is a constant if the <em>iterable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> denotes a constant, or if the Variable_Indexing aspect is not specified for the type of the <em>iterable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>; otherwise it is a variable.<br /></p>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>{"{"}<em>{"AI12-0093-1"}</em>{"}"} {"{"}<em>{"AI12-0156-1"}</em>{"}"} <br />The loop parameter of a generalized iterator has the same accessibility as the loop statement. This means that the loop parameter object is finalized when the loop statement is left. (It also may be finalized as part of assigning a new value to the loop parameter.) For array component iterators, the loop parameter directly denotes an element of the array and has the accessibility of the associated array. For container element iterators, the loop parameter denotes the result of the indexing function call (in the case of a constant indexing) or a generalized reference thereof (in the case of a variable indexing). Roughly speaking, the loop parameter has the accessibility level of a single iteration of the loop. More precisely, the function result (or the generalized reference thereof) is considered to be renamed in the declarative part of a notional block statement which immediately encloses the loop's <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code>; the accessibility of the loop parameter is that of the block statement. <br /></Admonition>


#### Dynamic Semantics

<p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} <br />For the execution of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with an <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code>, the <code><a href="../AA-5/AA-5.5#S0183">iterator_specification</a></code> is first elaborated. This elaboration elaborates the <code><a href="../AA-3/AA-3.2#S0027">subtype_indication</a></code>, if any.<br /></p>

<p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} {"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0266-1"}</em>{"}"} <br />For a sequential generalized iterator, the loop parameter is created, the <em>iterator_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is evaluated, and the denoted iterator object becomes the <em>loop iterator</em>. In a forward generalized iterator, the operation First of the iterator type is called on the loop iterator, to produce the initial value for the loop parameter. If the result of calling Has_Element on the initial value is False, then the execution of the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. Otherwise, the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is conditionally executed and then the Next operation of the iterator type is called with the loop iterator and the current value of the loop parameter to produce the next value to be assigned to the loop parameter. This repeats until the result of calling Has_Element on the loop parameter is False, or the loop is left as a consequence of a transfer of control. For a reverse generalized iterator, the operations Last and Previous are called rather than First and Next.<br /></p>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>{"{"}<em>{"AI12-0093-1"}</em>{"}"} <br />The loop parameter of a generalized iterator is a variable of which the user only has a constant view. It follows the normal rules for a variable of its nominal subtype. In particular, if the nominal subtype is indefinite, the variable is constrained by its initial value. Similarly, if the nominal subtype is class-wide, the variable (like all variables) has the tag of the initial value. Constraint_Error may be raised by a subsequent iteration if Next or Previous return an object with a different tag or constraint. <br /></Admonition>

<p>{"{"}<em>{"AI12-0266-1"}</em>{"}"} <br />For a parallel generalized iterator, the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>, if any, of the associated parallel construct, is first elaborated, to determine the maximum number of chunks (see 5.5), and then the operation Split_Into_Chunks of the iterator type is called, with the determined maximum passed as the Max_Chunks parameter, specifying the upper bound for the number of loop parameter objects (and the number of logical threads of control) to be associated with the iterator. In the absence of a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>, the maximum number of chunks is determined in an implementation-defined manner.<br /></p>

<Admonition type="aarm" aarm="implementation-defined" title="Implementation defined: ">
<strong></strong>The maximum number of chunks for a parallel generalized iterator without a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code>.<br /></Admonition>

<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<strong></strong>The Max_Chunks parameter of the Split_Into_Chunks procedure is an upper bound for the number of chunks to be associated with a loop. A container implementation may opt for a lower value for the number of chunks if a more optimal split can be determined. For instance, a tree-based container might create the split based on the number of branches at the top levels of the tree. <br /></Admonition>

<p>{"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0266-1"}</em>{"}"} {"{"}<em>{"AI12-0418-1"}</em>{"}"} <br />Upon return from Split_Into_Chunks, the actual number of chunks for the loop is determined by calling the Chunk_Count operation of the iterator, at which point one logical thread of control is initiated for each chunk, with an associated chunk index in the range from one to the actual number of chunks.<br /></p>

<p>{"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0266-1"}</em>{"}"} {"{"}<em>{"AI12-0418-1"}</em>{"}"} <br />Within each logical thread of control, a loop parameter is created. If a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> with a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> is present in the associated parallel construct, then a chunk parameter is created and initialized with a value from the discrete subtype defined by the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>, so that the order of the chosen chunk parameter values correspond to the order of the chunk indices associated with the logical threads of control. The operation First of the iterator type that has a Chunk parameter is called on the loop iterator, with Chunk initialized from the corresponding chunk index, to produce the initial value for the loop parameter. If the result of calling Has_Element on this initial value is False, then the execution of the logical thread of control is complete. Otherwise, the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is conditionally executed, and then the Next operation of the iterator type that has a Chunk parameter is called with the loop iterator, the current value of the loop parameter, and the corresponding chunk index, to produce the next value to be assigned to the loop parameter. This repeats until the result of calling Has_Element on the loop parameter is False, or the associated parallel construct is left as a consequence of a transfer of control.<br /></p>

<p>{"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0266-1"}</em>{"}"} {"{"}<em>{"AI12-0418-1"}</em>{"}"} <br />In the absence of a transfer of control, the associated parallel construct of a parallel generalized iterator is complete when all of its logical threads of control are complete.<br /></p>

<p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} {"{"}<em>{"AI05-0292-1"}</em>{"}"} {"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0266-1"}</em>{"}"} <br />For an array component iterator, the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> of the associated parallel construct, if any, is first elaborated to determine the maximum number of chunks (see 5.5), and then the <em>iterable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is evaluated and the denoted array object becomes the <em>array for the loop</em>. If the array for the loop is a null array, then the execution of the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. Otherwise, the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> is conditionally executed with the loop parameter denoting each component of the array for the loop, using a <em>canonical</em> order of components, which is last dimension varying fastest (unless the array has convention Fortran, in which case it is first dimension varying fastest). For a forward array component iterator, the iteration starts with the component whose index values are each the first in their index range, and continues in the canonical order. For a reverse array component iterator, the iteration starts with the component whose index values are each the last in their index range, and continues in the reverse of the canonical order. For a parallel array component iterator, the iteration is broken up into contiguous chunks of the canonical order, such that all components are covered with no overlaps; each chunk has its own logical thread of control with its own loop parameter and iteration within each chunk is in the canonical order. The number of chunks is implementation defined, but is limited in the presence of a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> to the determined maximum. The loop iteration proceeds until the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> has been conditionally executed for each component of the array for the loop, or until the loop is left as a consequence of a transfer of control.<br /></p>

<Admonition type="aarm" aarm="implementation-defined" title="Implementation defined: ">
<strong></strong>The number of chunks for an array component iterator.<br /></Admonition>

<p>{"{"}<em>{"AI12-0266-1"}</em>{"}"} <br />If a <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> with a <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code> is present in the associated parallel construct, then the logical thread of control associated with a given chunk has a chunk parameter initialized with a distinct value from the discrete subtype defined by the <code><a href="../AA-3/AA-3.6#S0055">discrete_subtype_definition</a></code>. The values of the chunk parameters are assigned such that they increase in the canonical order of the starting array components for the chunks.<br /></p>

<p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} {"{"}<em>{"AI05-0292-1"}</em>{"}"} {"{"}<em>{"AI12-0111-1"}</em>{"}"} {"{"}<em>{"AI12-0266-1"}</em>{"}"} <br />For a container element iterator, the <code><a href="../AA-5/AA-5.5#S0180">chunk_specification</a></code> of the associated parallel construct, if any, is first elaborated to determine the maximum number of chunks (see 5.5), and then the <em>iterable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is evaluated. If the container type has Iterator_View specified, an object of the Iterator_View type is created with the discriminant referencing the iterable container object denoted by the <em>iterable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>. This is the <em>iterable container object for the loop</em>. Otherwise, the iterable container object denoted by the <em>iterable_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> becomes the iterable container object for the loop. The default iterator function for the type of the iterable container object for the loop is called on the iterable container object and the result is the <em>loop iterator</em>. For a sequential container element iterator, an object of the default cursor subtype is created (the <em>loop cursor</em>). For a parallel container element iterator, each chunk of iterations will have its own loop cursor, again of the default cursor subtype.<br /></p>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>{"{"}<em>{"AI12-0111-1"}</em>{"}"} <br />If Iterator_View is specified, we add an extra object and use that object for this iteration. This allows these iterators to automatically use the stable view (defined in each of the language-defined containers) to do the iteration. That eliminates the need to set and clear the tampering with elements indication each time Reference is called; that eliminates substantial overhead as finalization is typically used to implement the tampering reset. <br /></Admonition>

<p>{"{"}<em>{"AI05-0139-2"}</em>{"}"} {"{"}<em>{"AI05-0292-1"}</em>{"}"} {"{"}<em>{"AI12-0266-1"}</em>{"}"} <br />A container element iterator then proceeds as described above for a generalized iterator, except that each reference to a loop parameter is replaced by a reference to the corresponding loop cursor. For a container element iterator, the loop parameter for each iteration instead denotes an indexing (see 4.1.6) into the iterable container object for the loop, with the only parameter to the indexing being the value of the loop cursor for the given iteration. If the loop parameter is a constant (see above), then the indexing uses the default constant indexing function for the type of the iterable container object for the loop; otherwise it uses the default variable indexing function.<br /></p>

<p>{"{"}<em>{"AI12-0120-1"}</em>{"}"} <br />Any exception propagated by the execution of a generalized iterator or container element iterator is propagated by the immediately enclosing loop statement.<br /></p>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>This text covers exceptions raised by called functions that make up the execution of the iterator as well as exceptions raised by the assignment to the loop parameter or cursor. <br /></Admonition>

#### Incompatibilities With Ada 2012

<Admonition type="aarm" aarm="note" title="Note: ">
{"{"}<em>{"AI12-0047-1"}</em>{"}"} <strong><br />Corrigendum:</strong> Added a rule to ensure that the object being iterated cannot be a component that could disappear before the loop completes. This could be incompatible by making a loop that was legal (and worked correctly, so long as the enclosing object is not modified during the loop) from the original Ada 2012 illegal in corrected Ada 2012. Such loops should be pretty rare, especially as these iterator forms are new to Ada 2012.<br /></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
{"{"}<em>{"AI12-0120-1"}</em>{"}"} <strong><br />Corrigendum:</strong> Added rules to reject loops if the call to the default iterator function for a container element iterator is illegal, or if the cursor type of an iterator is limited. These are formally incompatible with original Ada 2012, but as it's unlikely that any Ada 2012 compiler ever allowed the illegal usages in an expansion of a loop (it's much more likely that they would have just caused an internal error in the compiler), this should have no effect in practice.<br /></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
{"{"}<em>{"AI12-0151-1"}</em>{"}"} <strong><br />Corrigendum:</strong> Added a requirement that the given subtype statically match the subtype of the element or component for a component element iterator or array component iterator. Original Ada 2012 text allowed any type that covered the subtype of the element or component, but that led to questions of what the meaning was if they are different. In this case, the element is essentially a renaming of the container element, and it doesn't make sense for the constraints to be different. Ignoring explicitly defined constraints in renames is a mistake that we don't want to continue, thus we require static matching. This means that some programs might be illegal, but those programs were misleading at best, and potentially would raise unexpected exceptions because the element values might have been invalid or abnormal with respect to the declared constraint. <br /></Admonition>

#### Static Semantics

<p>{"{"}<em>{"AI12-0189-1"}</em>{"}"} {"{"}<em>{"AI12-0250-1"}</em>{"}"} {"{"}<em>{"AI12-0308-1"}</em>{"}"} {"{"}<em>{"AI12-0326-2"}</em>{"}"} <br />A <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with an <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> that has a <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code> is equivalent to a local declaration of a procedure P followed by a <code><a href="../AA-6/AA-6.4#S0217">procedure_call_statement</a></code> that is formed from the <code><a href="../AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> by replacing the {"<"}{">"} of the <code><a href="../AA-5/AA-5.5#S0190">parameter_association_with_box</a></code> with P'Access. The <code><a href="../AA-6/AA-6.1#S0206">formal_part</a></code> of the locally declared procedure P is formed from the <code><a href="../AA-6/AA-6.1#S0206">formal_part</a></code> of the anonymous access-to-procedure type <em>A</em>, by replacing the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> of each formal parameter of this <code><a href="../AA-6/AA-6.1#S0206">formal_part</a></code> with the <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> of the corresponding formal parameter or element of the list of <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code>s given in the <code><a href="../AA-5/AA-5.5#S0186">iterator_parameter_specification</a></code>. The body of <em>P</em> consists of the conditionally executed <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code>. The procedure P is called the <em>loop body procedure</em>.<br /></p>

<Admonition type="aarm" aarm="implementation-note" title="Implementation Note: ">
<strong></strong>{"{"}<em>{"AI12-0250-1"}</em>{"}"} <br />For a <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code> with an <code><a href="../AA-5/AA-5.5#S0182">iterator_filter</a></code>, the body of the routine would be something like:<br /></Admonition>

<CodeBlock>
procedure P ... is{"\n"}begin{"\n"}   if <a href="../AA-5/AA-5.5#S0182">iterator_filter</a> then{"\n"}      <a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a>{"\n"}   end if;{"\n"}end P;{"\n"}

</CodeBlock>
<p>{"{"}<em>{"AI12-0344-1"}</em>{"}"} <br />In a procedural iterator, the Parallel_Calls aspect (see 9.10.1) of the loop body procedure is True if the reserved word <strong>parallel</strong> occurs in the corresponding loop statement, and False otherwise.<br /></p>

<p>{"{"}<em>{"AI12-0189-1"}</em>{"}"} {"{"}<em>{"AI12-0326-2"}</em>{"}"} {"{"}<em>{"AI12-0344-1"}</em>{"}"} <br />The following aspects may be specified for a callable entity <em>S</em> that has exactly one formal parameter of an anonymous access-to-subprogram type:<br /></p>

<dt><br/>Allows_Exit</dt>
<dl>
<dd>The Allows_Exit aspect is of type Boolean. The specified value shall be static. The Allows_Exit aspect of an inherited primitive subprogram is True if Allows_Exit is True either for the corresponding subprogram of the progenitor type or for any other inherited subprogram that it overrides. If not specified or inherited as True, the Allows_Exit aspect of a callable entity is False. For an entry, only a confirming specification of False is permitted for the Allows_Exit aspect.<br /></dd>
</dl>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>An entry does not allow exit, because implementing a transfer of control out of a task or protected entry creates unnecessarily complex dynamic semantics. <br /></Admonition>

<dl>
<dd>Specifying the Allows_Exit aspect to be True for a subprogram indicates that the subprogram <em>allows exit</em>, meaning that it is prepared to be completed by arbitrary transfers of control from the loop body procedure[, including propagation of exceptions. A subprogram for which Allows_Exit is True should use finalization as appropriate rather than exception handling to recover resources and make any necessary final updates to data structures].<br /></dd>
</dl>
<Admonition type="aarm" aarm="note" title="Note: ">
<strong>Aspect Description for </strong><strong>Allows_Exit: </strong>An indication of whether a subprogram will operate correctly for arbitrary transfers of control.<br /></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>A subprogram that does not need cleanup satisfies the requirements, and thus can specify Allows_Exit as True. If a subprogram <em>S</em> allows exit, it cannot expect to get control other than via finalization if the loop body procedure initiates a transfer of control as part of a <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code>. In particular, exception handlers in <em>S</em>, even <strong>when others</strong> handlers, will not be executed when a transfer of control occurs. The mechanism that the implementation uses to implement such transfers of control needs to avoid triggering exception handlers. <br /></Admonition>

<dt><br/>Parallel_Iterator</dt>
<dl>
<dd>The Parallel_Iterator aspect is of type Boolean. The specified value shall be static. The Parallel_Iterator aspect of an inherited primitive subprogram is True if Parallel_Iterator is True either for the corresponding subprogram of the progenitor type or for any other inherited subprogram that it overrides. If not specified or inherited as True, the Parallel_Iterator aspect of a callable entity is False.<br /></dd>
<dd>{"{"}<em>{"AI12-0189-1"}</em>{"}"} {"{"}<em>{"AI12-0442-1"}</em>{"}"} <br />Specifying the Parallel_Iterator aspect to be True for a callable entity indicates that the entity is allowed to invoke the loop body procedure from multiple distinct logical threads of control. The Parallel_Iterator aspect for a subprogram shall be statically False if the subprogram allows exit.<br /></dd>
</dl>
<Admonition type="aarm" aarm="note" title="Note: ">
<strong>Aspect Description for </strong><strong>Parallel_Iterator: </strong>An indication of whether a subprogram may use multiple threads of control to invoke a loop body procedure.<br /></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>Permitting exit from a parallel procedural iterator introduces additional semantic and implementation complexity. <br /></Admonition>


#### Legality Rules

<p>{"{"}<em>{"AI12-0189-1"}</em>{"}"} {"{"}<em>{"AI12-0326-2"}</em>{"}"} <br />If a callable entity overrides an inherited dispatching subprogram that allows exit, the overriding callable entity also shall allow exit. If a callable entity overrides an inherited dispatching subprogram that has a True Parallel_Iterator aspect, the overriding callable entity also shall have a True Parallel_Iterator aspect.<br /></p>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>Since an entry never allows exit, attempting to implement an allows exit subprogram with a task or protected entry is always illegal. However, the Parallel_Iterator aspect can be applied to an entry, so a subprogram with the Parallel_Iterator aspect True can be implemented by an entry. <br /></Admonition>

<p>{"{"}<em>{"AI12-0326-2"}</em>{"}"} <br />A <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with a <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code> as its <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> shall begin with the reserved word <strong>parallel</strong> if and only if the callable entity identified in the <code><a href="../AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> has a Parallel_iterator aspect of True.<br /></p>

<p>{"{"}<em>{"AI12-0326-2"}</em>{"}"} <br />If the actual parameter of an anonymous access-to-subprogram type, passed in an explicit call of a subprogram for which the Parallel_Iterator aspect is True, is of the form <em>P</em>'Access, the designated subprogram <em>P</em> shall have a Parallel_Calls aspect True (see 9.10.1).<br /></p>

<p>{"{"}<em>{"AI12-0189-1"}</em>{"}"} {"{"}<em>{"AI12-0326-2"}</em>{"}"} <br />The <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with a <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code> as its <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> shall contain an <code><a href="../AA-5/AA-5.7#S0193">exit_statement</a></code>, return statement, <code><a href="../AA-5/AA-5.8#S0194">goto_statement</a></code>, or <code><a href="../AA-9/AA-9.5#S0265">requeue_statement</a></code> that leaves the loop only if the callable entity associated with the <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code> allows exit.<br /></p>

<p>{"{"}<em>{"AI12-0294-1"}</em>{"}"} <br />The <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with a <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code> as its <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> shall not contain an <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code> whose <code><a href="../AA-9/AA-9.5#S0257">entry_declaration</a></code> occurs outside the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>.<br /></p>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<strong></strong>An <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code> is not allowed in a procedure (see 9.5.2), it has to be directly in a <code><a href="../AA-9/AA-9.1#S0248">task_body</a></code>. Since the loop body here is implemented as  a procedure, we can't allow <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code>s there, either, even if the loop itself is directly in a <code><a href="../AA-9/AA-9.1#S0248">task_body</a></code>. <br /></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<strong></strong>This includes cases where the <code><a href="../AA-9/AA-9.5#S0258">accept_statement</a></code> is part of another construct, for instance, a <code><a href="../AA-9/AA-9.7#S0269">select_statement</a></code>. <br /></Admonition>


#### Dynamic Semantics

<p>{"{"}<em>{"AI12-0326-2"}</em>{"}"} {"{"}<em>{"AI12-0355-2"}</em>{"}"} {"{"}<em>{"AI12-0445-1"}</em>{"}"} <br />[For the execution of a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with an <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> that has a <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code>, the procedure denoted by the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> or <code><a href="../AA-4/AA-4.1#S0093">prefix</a></code> of the <code><a href="../AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> (the <em>iterating procedure</em>) is invoked, passing an access value designating the loop body procedure as a parameter. The iterating procedure then calls the loop body procedure zero or more times and returns, whereupon the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> is complete. If the <strong>parallel</strong> reserved word is present, the iterating procedure is allowed to invoke the loop body procedure from multiple distinct logical threads of control.] The <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>, if any, is elaborated prior to the invocation of the iterating procedure.<br /></p>

<Admonition type="aarm" aarm="proof" title="Proof: ">
<strong></strong>The stated dynamic semantics are implied by the static semantics given above and the bounded errors given below. <br /></Admonition>


#### Bounded (Run-Time) Errors

<p>{"{"}<em>{"AI12-0326-2"}</em>{"}"} {"{"}<em>{"AI12-0445-1"}</em>{"}"} <br />If the callable entity identified in the <code><a href="../AA-5/AA-5.5#S0187">iterator_procedure_call</a></code> allows exit, then it is a bounded error for a call of the loop body procedure to be performed from within an abort-deferred operation (see 9.8), unless the entire <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> was within the same abort-deferred operation. If detected, Program_Error is raised at the point of the call; otherwise, a transfer of control from the <code><a href="../AA-5/AA-5.1#S0166">sequence_of_statements</a></code> of the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> will not necessarily terminate the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>, and the loop body procedure can be called again.<br /></p>

<p>{"{"}<em>{"AI12-0326-2"}</em>{"}"} {"{"}<em>{"AI12-0445-1"}</em>{"}"} <br />If a <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code> with the <code><a href="../AA-5/AA-5.5#S0185">procedural_iterator</a></code> as its <code><a href="../AA-5/AA-5.5#S0179">iteration_scheme</a></code> (see 5.5) does not begin with the reserved word <strong>parallel</strong>, it is a bounded error if the loop body procedure is invoked from a different logical thread of control than the one that initiates the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>. If detected, Program_Error is raised; otherwise, conflicts associated with concurrent executions of the loop body procedure can occur without being detected by the applicable conflict check policy (see 9.10.1). Furthermore, propagating an exception or making an attempt to exit in the presence of multiple threads of control will not necessarily terminate the <code><a href="../AA-5/AA-5.5#S0178">loop_statement</a></code>, deadlock can occur, or the loop body procedure can be called again.<br /></p>

<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<strong></strong>{"{"}<em>{"AI12-0326-2"}</em>{"}"} <br />Other Ada rules are still in effect for the allows exit subprogram <em>A</em>, of course. For instance, if a transfer of control causes finalization which raises an exception, Program_Error will be propagated by <em>A</em> (rather than the transfer of control). In such a case, the bounded error above would still apply. Another example is the case where an unrelated task is waiting on the normal completion of the loop body procedure call in <em>A</em>. Such a task might end up waiting forever if a transfer of control happens (this is a deadlock situation). This case does not require additional wording, as the same thing would happen if an exception is propagated from the loop body procedure or if <em>A</em> executed a transfer of control (such as a return statement). <br /></Admonition>
