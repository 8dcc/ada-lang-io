<p>[A <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> declares another name for an entity, such as an object, exception, package, subprogram, entry, or generic unit. Alternatively, a <code><a href="../AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code> can be the completion of a previous <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>.]</p>
<Admonition type="aarm" aarm="glossary-entry" title="Glossary entry: ">
<p><strong></strong>A <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> is a declaration that does not define a new entity, but instead defines a view of an existing entity.</p></Admonition>

<p>Version=[5],Kind=(AddedNormal),Group=[C],Term=[renaming], Def=[a declaration that does not define a new entity, but instead defines a new view of an existing entity] </p>

#### Syntax

<p><code>renaming_declaration</code><a id="S0238"></a><code> ::= </code>      <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code>    | <code><a href="../AA-8/AA-8.5#S0240">exception_renaming_declaration</a></code>    | <code><a href="../AA-8/AA-8.5#S0241">package_renaming_declaration</a></code>    | <code><a href="../AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code>    | <code><a href="../AA-8/AA-8.5#S0243">generic_renaming_declaration</a></code></p>

#### Dynamic Semantics

<p>The elaboration of a <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> evaluates the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that follows the reserved word <strong>renames</strong> and thereby determines the view and entity denoted by this name (the <em>renamed view</em> and <em>renamed entity</em>). [A <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes the <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> denotes (a new view of) the renamed entity.] </p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 1   {"{"}<em>{"AI12-0442-1"}</em>{"}"} Renaming can be used to resolve name conflicts and to act as a shorthand. Renaming with a different <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> or <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code> does not hide the old <code><a href="../AA-4/AA-4.1#S0091">name</a></code>; the new <code><a href="../AA-4/AA-4.1#S0091">name</a></code> and the old <code><a href="../AA-4/AA-4.1#S0091">name</a></code> can be visible at different places.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p><em>This paragraph was deleted.</em>{"{"}<em>{"AI12-0427-1"}</em>{"}"} </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 2   A subtype defined without any additional constraint can be used to achieve the effect of renaming another subtype (including a task or protected subtype) as in </p></Admonition>

<CodeBlock>
   subtype Mode is Ada.Text_IO.File_Mode;{"\n"}
</CodeBlock>


#### Wording Changes from Ada 83

<Admonition type="aarm" aarm="note" title="Note: ">
<p>The second sentence of RM83-8.5(3), "At any point where a renaming declaration is visible, the identifier, or operator symbol of this declaration denotes the renamed entity." is incorrect. It doesn't say directly visible. Also, such an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> might resolve to something else.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>The verbiage about renamings being legal "only if exactly one...", which appears in RM83-8.5(4) (for objects) and RM83-8.5(7) (for subprograms) is removed, because it follows from the normal rules about overload resolution. For language lawyers, these facts are obvious; for programmers, they are irrelevant, since failing these tests is highly unlikely. </p></Admonition>


## 8.5.1  Object Renaming Declarations

<p>{"{"}<em>{"AI12-0383-1"}</em>{"}"} [An <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code> is used to rename an object or value.] </p>

#### Syntax

<p>{"{"}<em>{"AI95-00230-01"}</em>{"}"} {"{"}<em>{"AI95-00423-01"}</em>{"}"} {"{"}<em>{"AI05-0183-1"}</em>{"}"} {"{"}<em>{"AI12-0275-1"}</em>{"}"} <code>object_renaming_declaration</code><a id="S0239"></a><code> ::= </code>    <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> [: [<code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>] <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>] <strong>renames</strong> <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];  | <code><a href="../AA-3/AA-3.1#S0022">defining_identifier</a></code> : <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> <strong>renames</strong> <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];</p>

#### Name Resolution Rules

<p>{"{"}<em>{"AI95-00230-01"}</em>{"}"} {"{"}<em>{"AI95-00254-01"}</em>{"}"} {"{"}<em>{"AI95-00409-01"}</em>{"}"} {"{"}<em>{"AI12-0275-1"}</em>{"}"} The type of the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall resolve to the type determined by the <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>, if present. If no <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> or <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> is present, the expected type of the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is any type.</p>
<p>{"{"}<em>{"AI12-0275-1"}</em>{"}"} In the case where the type is defined by an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, the type of the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall resolve to an anonymous access type. If the anonymous access type is an access-to-object type, the type of the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall have the same designated type as that of the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>. If the anonymous access type is an access-to-subprogram type, the type of the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall have a designated profile that is type conformant with that of the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>.</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>A previous version of Ada 9X used the usual "expected type" wording:"The expected type for the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is that determined by the <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code>."We changed it so that this would be illegal: </p></Admonition>

<CodeBlock>
X: T;{"\n"}Y: T'Class renames X; -- Illegal!{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>When the above was legal, it was unclear whether Y was of type T or T'Class. Note that we still allow this: </p></Admonition>

<CodeBlock>
Z: T'Class := ...;{"\n"}W: T renames F(Z);{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>where F is a function with a controlling parameter and result. This is admittedly a bit odd.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>Note that the matching rule for generic formal parameters of mode <strong>in out</strong> was changed to keep it consistent with the rule for renaming. That makes the rule different for <strong>in</strong> vs. <strong>in out</strong>.</p></Admonition>


#### Legality Rules

<p>{"{"}<em>{"AI12-0383-1"}</em>{"}"} The renamed entity shall be an object or value.</p>
<p>{"{"}<em>{"AI95-00231-01"}</em>{"}"} {"{"}<em>{"AI95-00409-01"}</em>{"}"} {"{"}<em>{"AI12-0383-1"}</em>{"}"} In the case where the type is defined by an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>, the type of the renamed entity and the type defined by the <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code>: </p>
<p>{"{"}<em>{"AI95-00231-01"}</em>{"}"} {"{"}<em>{"AI95-00409-01"}</em>{"}"} shall both be access-to-object types with statically matching designated subtypes and with both or neither being access-to-constant types; or </p>
<p>{"{"}<em>{"AI95-00409-01"}</em>{"}"} shall both be access-to-subprogram types with subtype conformant designated profiles. </p>
<p>{"{"}<em>{"AI95-00423-01"}</em>{"}"} {"{"}<em>{"AI12-0287-1"}</em>{"}"} For an <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code> with a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code> or an <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> that has a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>, the subtype of the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> shall exclude null. In addition, if the <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code> occurs within the body of a generic unit <em>G</em> or within the body of a generic unit declared within the declarative region of generic unit <em>G</em>, then:</p>
<p>{"{"}<em>{"AI12-0287-1"}</em>{"}"} if the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> statically denotes a generic formal object of mode <strong>in out</strong> of <em>G</em>, then the declaration of that object shall have a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>;</p>
<p>{"{"}<em>{"AI12-0287-1"}</em>{"}"} if the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> statically denotes a call of a generic formal function of <em>G</em>, then the declaration of the result of that function shall have a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>. </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI12-0287-1"}</em>{"}"} {"{"}<em>{"AI12-0005-1"}</em>{"}"} These rules prevent "lying". <strong>Null</strong> must never be the value of an object with an explicit <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>. The bullets are assume-the-worst rules that prevent trouble in two obscure cases: </p></Admonition>

<CodeBlock>
type Acc_I is access Integer;{"\n"}subtype Acc_NN_I is not null Acc_I;{"\n"}Obj : Acc_I := null;{"\n"}
</CodeBlock>

<CodeBlock>
generic{"\n"}   B : in out Acc_NN_I;{"\n"}package Gen is{"\n"}   ...{"\n"}end Gen;{"\n"}
</CodeBlock>

<CodeBlock>
package body Gen is{"\n"}   D : not null Acc_I renames B;{"\n"}end Gen;{"\n"}
</CodeBlock>

<CodeBlock>
package Inst is new Gen (B ={">"} Obj);{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI12-0287-1"}</em>{"}"} Without the first bullet rule, D would be legal, and contain the value <strong>null</strong>, because the rule about lying is satisfied for generic matching (Obj matches B; B does not explicitly state <strong>not null</strong>), Legality Rules are not rechecked in the body of any instance, and the template passes the lying rule as well. The second bullet handles a similar case involving formal functions. The rules are so complex because they have to apply to formals used in bodies of child generics as well as in the bodies of generics. </p></Admonition>

<p>{"{"}<em>{"AI12-0401-1"}</em>{"}"} In the case where the <em>object_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is a <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code> with a nominal subtype <em>S</em> and whose <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is a <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes an object <em>Q</em>: </p>
<p>if <em>S</em> is an elementary subtype, then: </p>
<p><em>Q</em> shall be a constant other than a dereference of an access type; or</p>
<p>the nominal subtype of <em>Q</em> shall be statically compatible with <em>S</em>; or</p>
<p><em>S</em> shall statically match the base subtype of its type if scalar, or the first subtype of its type if an access type. </p>
<p>if <em>S</em> is a composite subtype, then <em>Q</em> shall be known to be constrained or <em>S</em> shall statically match the first subtype of its type. </p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>There's no restriction if the <code><a href="../AA-4/AA-4.4#S0132">expression</a></code> is a value. </p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>This check prevents the renamed object from violating its nominal subtype. As the subtype is only checked when the object is renamed, we make it illegal if the actual object is a variable whose value could be changed afterwards to violate the subtype. This is messy as "known to be constrained" is only defined for composite objects, so we have to handle elementary objects and all values separately. </p></Admonition>

<p>{"{"}<em>8652/0017</em>{"}"} {"{"}<em>{"AI95-00184-01"}</em>{"}"} {"{"}<em>{"AI95-00363-01"}</em>{"}"} {"{"}<em>{"AI05-0008-1"}</em>{"}"} {"{"}<em>{"AI12-0401-1"}</em>{"}"} The renamed entity shall not be a subcomponent that depends on discriminants of an object whose nominal subtype is unconstrained unless the object is known to be constrained. A <code><a href="../AA-4/AA-4.1#S0097">slice</a></code> of an array shall not be renamed if this restriction disallows renaming of the array. </p>
<p>{"{"}<em>{"AI12-0401-1"}</em>{"}"} In addition to the places where Legality Rules normally apply (see 12.3), these rules also apply in the private part of an instance of a generic unit.</p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>This applies to all of the Legality Rules in this subclause. Rechecks are needed for most of the rules (but not the first two). </p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>This prevents renaming of subcomponents that might disappear, which might leave dangling references. Similar restrictions exist for the Access attribute.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>8652/0017</em>{"}"} {"{"}<em>{"AI95-00184-01"}</em>{"}"} {"{"}<em>{"AI05-0008-1"}</em>{"}"} The "recheck on instantiation" requirement on generics is necessary to avoid renaming of components which could disappear even when the nominal subtype would prevent the problem:</p></Admonition>

<CodeBlock>
type T1 (D1 : Boolean) is{"\n"}   record{"\n"}      case D1 is{"\n"}         when False ={">"}{"\n"}            C1 : Integer;{"\n"}         when True ={">"}{"\n"}            null;{"\n"}         end case;{"\n"}      end record;{"\n"}
</CodeBlock>

<CodeBlock>
generic{"\n"}   type F is new T1;{"\n"}   X : in out F;{"\n"}package G is{"\n"}   C1_Ren : Integer renames X.C1;{"\n"}end G;{"\n"}
</CodeBlock>

<CodeBlock>
type T2 (D2 : Boolean := False) is new T1 (D1 ={">"} D2);{"\n"}{"\n"}Y : T2;{"\n"}{"\n"}package I is new G (T2, Y);{"\n"}{"\n"}Y := (D1 ={">"} True); -- Oops!  What happened to I.C1_Ren?{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI05-0008-1"}</em>{"}"} In addition, the "known to be constrained" rules include assume-the-worst rules for generic bodies partially to prevent such problems.</p></Admonition>

<Admonition type="aarm" aarm="implementation-note" title="Implementation Note: ">
<p><strong></strong>Note that if an implementation chooses to deallocate-then-reallocate on <code><a href="../AA-5/AA-5.2#S0173">assignment_statement</a></code>s assigning to unconstrained definite objects, then it cannot represent renamings and access values as simple addresses, because the above rule does not apply to all components of such an object. </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>If it is a generic formal object, then the assume-the-best or assume-the-worst rules are applied as appropriate. </p></Admonition>


#### Static Semantics

<p>{"{"}<em>{"AI95-00230-01"}</em>{"}"} {"{"}<em>{"AI95-00409-01"}</em>{"}"} {"{"}<em>{"AI12-0383-1"}</em>{"}"} An <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code> declares a new view [of the renamed entity] whose properties are identical to those of the renamed view. [Thus, the properties of the renamed entity are not affected by the <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code>. In particular, its nominal subtype, whether it is a value or an object, its value if it is an object, and whether or not it is a constant, are unaffected; similarly, the constraints and other properties of its nominal subtype are not affected by renaming (any constraint implied by the <code><a href="../AA-3/AA-3.2#S0028">subtype_mark</a></code> or <code><a href="../AA-3/AA-3.10#S0084">access_definition</a></code> of the <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code> is ignored).] </p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>Because the constraints are ignored, it is a good idea to use the nominal subtype of the renamed object when writing an <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code>.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI95-00409-01"}</em>{"}"} If no <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code> is given in the renaming, the object may or may not exclude null. This is similar to the way that constraints need not match, and <strong>constant</strong> is not specified. The renaming defines a view of the renamed entity, inheriting the original properties. </p></Admonition>


#### Examples

<p><em>Example of renaming an object:</em> </p>
<CodeBlock>
declare{"\n"}   L : Person renames Leftmost_Person; -- see 3.10.1{"\n"}begin{"\n"}   L.Age := L.Age + 1;{"\n"}end;{"\n"}
</CodeBlock>

<p>{"{"}<em>{"AI12-0383-1"}</em>{"}"} Example of renaming a value:</p>
<CodeBlock>
{"{"}{"AI12-0383-1"}{"}"} Uno renames One;  -- see 3.3.2{"\n"}
</CodeBlock>


#### Wording Changes from Ada 83

<Admonition type="aarm" aarm="note" title="Note: ">
<p>The phrase "subtype ... as defined in a corresponding object declaration, component declaration, or component subtype indication", from RM83-8.5(5), is incorrect in Ada 95; therefore we removed it. It is incorrect in the case of an object with an indefinite unconstrained nominal subtype. </p></Admonition>


#### Incompatibilities With Ada 95

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI95-00363-01"}</em>{"}"} Aliased variables are not necessarily constrained in Ada 2005 (see 3.6). Therefore, a subcomponent of an aliased variable may disappear or change shape, and renaming such a subcomponent thus is illegal, while the same operation would have been legal in Ada 95. Note that most allocated objects are still constrained by their initial value (see 4.8), and thus have no change in the legality of renaming for them. For example, using the type T2 of the previous example: </p></Admonition>

<CodeBlock>
   AT2 : aliased T2;{"\n"}   C1_Ren : Integer renames AT2.C1; -- Illegal in Ada 2005, legal in Ada 95{"\n"}   AT2 := (D1 ={">"} True);             -- Raised Constraint_Error in Ada 95,{"\n"}                                    -- but does not in Ada 2005, so C1_Ren becomes{"\n"}                                    -- invalid when this is assigned.{"\n"}
</CodeBlock>


#### Extensions to Ada 95

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI95-00230-01"}</em>{"}"} {"{"}<em>{"AI95-00231-01"}</em>{"}"} {"{"}<em>{"AI95-00254-01"}</em>{"}"} {"{"}<em>{"AI95-00409-01"}</em>{"}"} A renaming can have an anonymous access type. In that case, the accessibility of the renaming is that of the original object (accessibility is not lost as it is for assignment to a component or stand-alone object).</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI95-00231-01"}</em>{"}"} {"{"}<em>{"AI95-00423-01"}</em>{"}"} A renaming can have a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>; if so, the renamed object must also exclude null, so that the <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code> does not lie. On the other hand, if the renaming does not have a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>. it excludes null if the renamed object does. </p></Admonition>

#### Incompatibilities With Ada 2005

<Admonition type="aarm" aarm="correction" title="Correction:">
<p>{"{"}<em>{"AI05-0008-1"}</em>{"}"} <strong></strong> Simplified the description of when a discriminant-dependent component is allowed to be renamed - it's now simply when the object is known to be constrained. This fixes a confusion as to whether a subcomponent of an object that is not certain to be constrained can be renamed. The fix introduces an incompatibility, as the rule did not apply in Ada 95 if the prefix was a constant; but it now applies no matter what kind of object is involved. The incompatibility is not too bad, since most kinds of constants are known to be constrained. </p></Admonition>


#### Extensions to Ada 2005

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI05-0183-1"}</em>{"}"} An optional <code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code> can be used in an <code><a href="../AA-8/AA-8.5#S0239">object_renaming_declaration</a></code>. This is described in 13.1.1. </p></Admonition>


#### Incompatibilities With Ada 2012

<Admonition type="aarm" aarm="correction" title="Correction:">
<p>{"{"}<em>{"AI12-0287-1"}</em>{"}"} <strong></strong> The Legality Rules for renames with null exclusions no longer applies to generic formal objects of mode <strong>in</strong>, but does apply to renames of generic formal functions. This means a few unlikely programs are now illegal that were previously allowed by original Ada 2012, while more programs that were previously llegal will be allowed.</p></Admonition>

<Admonition type="aarm" aarm="correction" title="Correction:">
<p>{"{"}<em>{"AI12-0401-1"}</em>{"}"} <strong></strong> Added a rule to ensure that a renaming of a <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code> of a variable is allowed only if the variable will always remain within the nominal subtype of the <code><a href="../AA-4/AA-4.7#S0163">qualified_expression</a></code>. This was not required in Ada 2012. Renamings that are now illegal are at risk of causing erroneous execution if the variable value is changed to a bad value; this is consistent with other rules preventing renamings from changing to violate their known properties. </p></Admonition>


#### Extensions to Ada 2012

## 8.5.4  Subprogram Renaming Declarations

<p>{"{"}<em>{"AI05-0299-1"}</em>{"}"} A <code><a href="../AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code> can serve as the completion of a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>; such a <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> is called a <em>renaming-as-body</em>. A <code><a href="../AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code> that is not a completion is called a <em>renaming-as-declaration</em>[, and is used to rename a subprogram (possibly an enumeration literal) or an entry]. </p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>{"{"}<em>{"AI05-0299-1"}</em>{"}"} A renaming-as-body is a declaration, as defined in Clause 3. </p></Admonition>


#### Syntax

<p>{"{"}<em>{"AI95-00218-03"}</em>{"}"} {"{"}<em>{"AI05-0183-1"}</em>{"}"} <code>subprogram_renaming_declaration</code><a id="S0242"></a><code> ::= </code>    [<code><a href="../AA-8/AA-8.3#S0234">overriding_indicator</a></code>]    <code><a href="../AA-6/AA-6.1#S0196">subprogram_specification</a></code> <strong>renames</strong> <em>callable_entity_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];</p>

#### Name Resolution Rules

<p>The expected profile for the <em>callable_entity_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> is the profile given in the <code><a href="../AA-6/AA-6.1#S0196">subprogram_specification</a></code>. </p>

#### Legality Rules

<p>{"{"}<em>{"AI05-0239-1"}</em>{"}"} {"{"}<em>{"AI12-0373-1"}</em>{"}"} The profile of a renaming-as-declaration shall be mode conformant with that of the renamed callable entity. </p>
<p>{"{"}<em>{"AI95-00423-01"}</em>{"}"} For a parameter or result subtype of the <code><a href="../AA-6/AA-6.1#S0196">subprogram_specification</a></code> that has an explicit <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>:</p>
<p>{"{"}<em>{"AI12-0287-1"}</em>{"}"} if the <em>callable_entity_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> statically denotes a generic formal subprogram of a generic unit <em>G</em>, and the <code><a href="../AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code> occurs within the body of a generic unit <em>G</em> or within the body of a generic unit declared within the declarative region of the generic unit <em>G</em>, then the corresponding parameter or result subtype of the formal subprogram of <em>G</em> shall have a <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>;</p>
<p>otherwise, the subtype of the corresponding parameter or result type of the renamed callable entity shall exclude null. In addition to the places where Legality Rules normally apply (see 12.3), this rule applies also in the private part of an instance of a generic unit. </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>This rule prevents "lying". <strong>Null</strong> must never be the value of a parameter or result with an explicit <code><a href="../AA-3/AA-3.10#S0083">null_exclusion</a></code>. The first bullet is an assume-the-worst rule which prevents trouble in generic bodies (including bodies of child units) when the formal subtype excludes null implicitly. </p></Admonition>

<p>{"{"}<em>8652/0027</em>{"}"} {"{"}<em>8652/0028</em>{"}"} {"{"}<em>{"AI95-00135-01"}</em>{"}"} {"{"}<em>{"AI95-00145-01"}</em>{"}"} {"{"}<em>{"AI05-0239-1"}</em>{"}"} The profile of a renaming-as-body shall conform fully to that of the declaration it completes. If the renaming-as-body completes that declaration before the subprogram it declares is frozen, the profile shall be mode conformant with that of the renamed callable entity and the subprogram it declares takes its convention from the renamed subprogram; otherwise, the profile shall be subtype conformant with that of the renamed callable entity and the convention of the renamed subprogram shall not be Intrinsic. A renaming-as-body is illegal if the declaration occurs before the subprogram whose declaration it completes is frozen, and the renaming renames the subprogram itself, through one or more subprogram renaming declarations, none of whose subprograms has been frozen. </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>The otherwise part of the second sentence is to allow an implementation of a renaming-as-body as a single jump instruction to the target subprogram. Among other things, this prevents a subprogram from being completed with a renaming of an entry. (In most cases, the target of the jump can be filled in at link time. In some cases, such as a renaming of a name like "A(I).<strong>all</strong>", an indirect jump is needed. Note that the name is evaluated at renaming time, not at call time.)</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>8652/0028</em>{"}"} {"{"}<em>{"AI95-00145-01"}</em>{"}"} The first part of the second sentence is intended to allow renaming-as-body of predefined operators before the <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> is frozen. For some types (such as integer types), the parameter type for operators is the base type, and it would be very strange for   <strong>function</strong> Equal (A, B : <strong>in</strong> T) <strong>return</strong> Boolean;   <strong>function</strong> Equal (A, B : <strong>in</strong> T) <strong>return</strong> Boolean <strong>renames</strong> "=";to be illegal. (Note that predefined operators cannot be renamed this way after the <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code> is frozen, as they have convention Intrinsic.)</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>The first sentence is the normal rule for completions of <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>s. </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>An <code><a href="../AA-9/AA-9.5#S0257">entry_declaration</a></code>, unlike a <code><a href="../AA-6/AA-6.1#S0195">subprogram_declaration</a></code>, cannot be completed with a <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code>. Nor can a <code><a href="../AA-12/AA-12.1#S0311">generic_subprogram_declaration</a></code>.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>The syntax rules prevent a protected subprogram declaration from being completed by a renaming. This is fortunate, because it allows us to avoid worrying about whether the implicit protected object parameter of a protected operation is involved in the conformance rules. </p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>8652/0027</em>{"}"} {"{"}<em>{"AI95-00135-01"}</em>{"}"} Circular renames before freezing is illegal, as the compiler would not be able to determine the convention of the subprogram. Other circular renames are handled below; see Bounded (Run-Time) Errors. </p></Admonition>

<p>{"{"}<em>{"AI95-00228-01"}</em>{"}"} The <em>callable_entity_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> of a renaming shall not denote a subprogram that requires overriding (see 3.9.3). </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI95-00228-01"}</em>{"}"} Such a rename cannot be of the inherited subprogram (which requires overriding because it cannot be called), and thus cannot squirrel away a subprogram (see below). That would be confusing, so we make it illegal. The renaming is allowed after the overriding, as then the <code><a href="../AA-4/AA-4.1#S0091">name</a></code> will denote the overriding subprogram, not the inherited one. </p></Admonition>

<p>{"{"}<em>{"AI95-00228-01"}</em>{"}"} The <em>callable_entity_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> of a renaming-as-body shall not denote an abstract subprogram. </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI95-00228-01"}</em>{"}"} Such a subprogram has no body, so it hardly can replace one in the program. </p></Admonition>

<p>{"{"}<em>{"AI12-0204-1"}</em>{"}"} If the <em>callable_entity_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code> of a renaming is a prefixed view, the prefix of that view shall denote an object for which renaming is allowed.</p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>{"AI12-0204-1"}</em>{"}"} The prefix in such a case is essentially renamed and passed to any calls of the renamed subprogram. If the prefix isn't legal to rename, that doesn't make sense (and allowing it might end up passing a nonexistent object to some calls). </p></Admonition>

<p>A <code><a href="../AA-4/AA-4.1#S0091">name</a></code> that denotes a formal parameter of the <code><a href="../AA-6/AA-6.1#S0196">subprogram_specification</a></code> is not allowed within the <em>callable_entity_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>. </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>This is to prevent things like this: </p></Admonition>

<CodeBlock>
function F(X : Integer) return Integer renames Table(X).all;{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>A similar rule in 6.1 forbids things like this: </p></Admonition>

<CodeBlock>
function F(X : Integer; Y : Integer := X) return Integer;{"\n"}
</CodeBlock>


#### Static Semantics

<p>A renaming-as-declaration declares a new view of the renamed entity. The profile of this new view takes its subtypes, parameter modes, and calling convention from the original profile of the callable entity, while taking the formal parameter <code><a href="../AA-4/AA-4.1#S0091">name</a></code>s and <code><a href="../AA-3/AA-3.7#S0063">default_expression</a></code>s from the profile given in the <code><a href="../AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code>. The new view is a function or procedure, never an entry. </p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p><strong>To be honest: </strong>When renaming an entry as a procedure, the compile-time rules apply as if the new view is a procedure, but the run-time semantics of a call are that of an entry call. </p></Admonition>

<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>For example, it is illegal for the <code><a href="../AA-9/AA-9.5#S0264">entry_call_statement</a></code> of a <code><a href="../AA-9/AA-9.7#S0276">timed_entry_call</a></code> to call the new view. But what looks like a procedure call will do things like barrier waiting.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>8652/0105</em>{"}"} {"{"}<em>{"AI95-00211-01"}</em>{"}"} {"{"}<em>{"AI95-00228-01"}</em>{"}"} {"{"}<em>{"AI05-0095-1"}</em>{"}"} All properties of the renamed entity are inherited by the new view unless otherwise stated by this document. In particular, if the renamed entity is abstract, the new view also is abstract. Similarly, if the renamed entity is not a program unit, then neither is the renaming. (Implicitly declared subprograms are not program units, see 10.1). </p></Admonition>


#### Dynamic Semantics

<p>{"{"}<em>8652/0014</em>{"}"} {"{"}<em>{"AI95-00064-01"}</em>{"}"} For a call to a subprogram whose body is given as a renaming-as-body, the execution of the renaming-as-body is equivalent to the execution of a <code><a href="../AA-6/AA-6.3#S0216">subprogram_body</a></code> that simply calls the renamed subprogram with its formal parameters as the actual parameters and, if it is a function, returns the value of the call. </p>
<Admonition type="aarm" aarm="ramification" title="Ramification: ">
<p><strong></strong>This implies that the subprogram completed by the renaming-as-body has its own elaboration check. </p></Admonition>

<p>{"{"}<em>{"AI05-0123-1"}</em>{"}"} For a call on a renaming of a dispatching subprogram that is overridden, if the overriding occurred before the renaming, then the body executed is that of the overriding declaration, even if the overriding declaration is not visible at the place of the renaming; otherwise, the inherited or predefined subprogram is called. A corresponding rule applies to a call on a renaming of a predefined equality operator for an untagged record type. </p>
<Admonition type="aarm" aarm="discussion" title="Discussion: ">
<p><strong></strong>Note that whether or not the renaming is itself primitive has nothing to do with the renamed subprogram.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI05-0123-1"}</em>{"}"} Note that the above rule is only for tagged types and equality of untagged record types.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>Consider the following example: </p></Admonition>

<CodeBlock>
package P is{"\n"}    type T is tagged null record;{"\n"}    function Predefined_Equal(X, Y : T) return Boolean renames "=";{"\n"}private{"\n"}    function "="(X, Y : T) return Boolean; -- Override predefined "=".{"\n"}end P;{"\n"}
</CodeBlock>

<CodeBlock>
with P; use P;{"\n"}package Q is{"\n"}    function User_Defined_Equal(X, Y : T) return Boolean renames P."=";{"\n"}end Q;{"\n"}
</CodeBlock>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>A call on Predefined_Equal will execute the predefined equality operator of T, whereas a call on User_Defined_Equal will execute the body of the overriding declaration in the private part of P.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>Thus a renaming allows one to squirrel away a copy of an inherited or predefined subprogram before later overriding it. </p></Admonition>


#### Bounded (Run-Time) Errors

<p>{"{"}<em>8652/0027</em>{"}"} {"{"}<em>{"AI95-00135-01"}</em>{"}"} If a subprogram directly or indirectly renames itself, then it is a bounded error to call that subprogram. Possible consequences are that Program_Error or Storage_Error is raised, or that the call results in infinite recursion. </p>
<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>{"{"}<em>8652/0027</em>{"}"} {"{"}<em>{"AI95-00135-01"}</em>{"}"} This has to be a bounded error, as it is possible for a renaming-as-body appearing in a package body to cause this problem. Thus it is not possible in general to detect this problem at compile time. </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 1   A procedure can only be renamed as a procedure. A function whose <code><a href="../AA-6/AA-6.1#S0200">defining_designator</a></code> is either an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> or an <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code> can be renamed with either an <code><a href="../AA-2/AA-2.3#S0002">identifier</a></code> or an <code><a href="../AA-6/AA-6.1#S0202">operator_symbol</a></code>; for renaming as an operator, the subprogram specification given in the <code><a href="../AA-8/AA-8.5#S0238">renaming_declaration</a></code> is subject to the rules given in 6.6 for operator declarations. Enumeration literals can be renamed as functions; similarly, <code><a href="../AA-4/AA-4.1#S0100">attribute_reference</a></code>s that denote functions (such as references to Succ and Pred) can be renamed as functions. An entry can only be renamed as a procedure; the new <code><a href="../AA-4/AA-4.1#S0091">name</a></code> is only allowed to appear in contexts that allow a procedure <code><a href="../AA-4/AA-4.1#S0091">name</a></code>. An entry of a family can be renamed, but an entry family cannot be renamed as a whole.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 2   The operators of the root numeric types cannot be renamed because the types in the profile are anonymous, so the corresponding specifications cannot be written; the same holds for certain attributes, such as Pos.</p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p><em>This paragraph was deleted.</em>{"{"}<em>{"AI12-0292-1"}</em>{"}"} </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 3   The primitiveness of a renaming-as-declaration is determined by its profile, and by where it occurs, as for any declaration of (a view of) a subprogram; primitiveness is not determined by the renamed view. In order to perform a dispatching call, the subprogram name has to denote a primitive subprogram, not a nonprimitive renaming of a primitive subprogram. </p></Admonition>

<Admonition type="aarm" aarm="reason" title="Reason: ">
<p><strong></strong>A <code><a href="../AA-8/AA-8.5#S0242">subprogram_renaming_declaration</a></code> could more properly be called <code>renaming_as_subprogram_declaration</code>, since you're renaming something as a subprogram, but you're not necessarily renaming a subprogram. But that's too much of a mouthful. Or, alternatively, we could call it a <code>callable_entity_renaming_declaration</code>, but that's even worse. Not only is it a mouthful, it emphasizes the entity being renamed, rather than the new view, which we think is a bad idea. We'll live with the oddity. </p></Admonition>

#### Inconsistencies With Ada 2005

<Admonition type="aarm" aarm="note" title="Note: ">
<p>{"{"}<em>{"AI05-0123-1"}</em>{"}"} Renaming of user-defined untagged record equality is now defined to call the overridden body so long as the overriding occurred before the renames. This could change the body called in unusual cases; the change is necessary to preserve the principle that the body called for an explicit call to "=" (via a renames in this case) is the same as the one inherited for a derived type and used in generics. Note that any renamings before the overriding will be unchanged. Any differences caused by the change will be rare and most likely will fix a bug. </p></Admonition>

#### Incompatibilities With Ada 2012

<Admonition type="aarm" aarm="correction" title="Correction:">
<p>{"{"}<em>{"AI12-0204-1"}</em>{"}"} <strong></strong> Added a rule to ensure that the prefix of a renaming of a prefixed view continues to exist during the life of a renames. If the prefix is a subcomponent that depends on discriminants, Ada 2005 and 2012 would have allowed the prefix while Ada 2022 would not. Without this change, explicit forms (renaming the object and then using that in calls) would be safer than the renaming; that's inconsistent with other kinds of renaming. </p></Admonition>

## 8.5.5  Generic Renaming Declarations

<p>[A <code><a href="../AA-8/AA-8.5#S0243">generic_renaming_declaration</a></code> is used to rename a generic unit.] </p>

#### Syntax

<p>{"{"}<em>{"AI05-0183-1"}</em>{"}"} <code>generic_renaming_declaration</code><a id="S0243"></a><code> ::= </code>    <strong>generic package</strong>	<code><a href="../AA-6/AA-6.1#S0201">defining_program_unit_name</a></code> <strong>renames</strong> <em>generic_package_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];  | <strong>generic procedure</strong>	<code><a href="../AA-6/AA-6.1#S0201">defining_program_unit_name</a></code> <strong>renames</strong> <em>generic_procedure_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];  | <strong>generic function</strong>	<code><a href="../AA-6/AA-6.1#S0201">defining_program_unit_name</a></code> <strong>renames</strong> <em>generic_function_</em><code><a href="../AA-4/AA-4.1#S0091">name</a></code>        [<code><a href="../AA-13/AA-13.1#S0346">aspect_specification</a></code>];</p>

#### Legality Rules

<p>The renamed entity shall be a generic unit of the corresponding kind. </p>

#### Static Semantics

<p>A <code><a href="../AA-8/AA-8.5#S0243">generic_renaming_declaration</a></code> declares a new view [of the renamed generic unit]. </p>
<Admonition type="aarm" aarm="note" title="Note: ">
<p>NOTE 1   {"{"}<em>{"AI12-0440-1"}</em>{"}"} Although the properties of the new view are the same as those of the renamed view, the place where the <code><a href="../AA-8/AA-8.5#S0243">generic_renaming_declaration</a></code> occurs can affect the legality of subsequent renamings and instantiations that denote the <code><a href="../AA-8/AA-8.5#S0243">generic_renaming_declaration</a></code>, in particular if the renamed generic unit is a library unit (see 10.1.1). </p></Admonition>


#### Examples

<p><em>Example of renaming a generic unit:</em> </p>
<CodeBlock>
generic package Enum_IO renames Ada.Text_IO.Enumeration_IO;  -- see A.10.10{"\n"}
</CodeBlock>


#### Extensions to Ada 83

<Admonition type="aarm" aarm="note" title="Note: ">
<p>Renaming of generic units is new to Ada 95. It is particularly important for renaming child library units that are generic units. For example, it might be used to rename Numerics.Generic_Elementary_Functions as simply Generic_Elementary_Functions, to match the name for the corresponding Ada-83-based package. </p></Admonition>

<Admonition type="aarm" aarm="note" title="Note: ">
