<p>A task unit is declared by a <em>task declaration</em>, which has a corresponding <code><a href="./AA-9.1#S0248">task_body</a></code>. A task declaration may be a <code><a href="./AA-9.1#S0244">task_type_declaration</a></code>, in which case it declares a named task type; alternatively, it may be a <code><a href="./AA-9.1#S0245">single_task_declaration</a></code>, in which case it defines an anonymous task type, as well as declaring a named task object of that type. </p>

#### Syntax

#### Static Semantics

<p>A <code><a href="./AA-9.1#S0246">task_definition</a></code> defines a task type and its first subtype. The first list of <code><a href="./AA-9.1#S0247">task_item</a></code>s of a <code>[task_definition](./AA-9.1#S0246)</code>, together with the <code>[known_discriminant_part](./AA-3.7#S0061)</code>, if any, is called the visible part of the task unit. [ The optional list of <code><a href="./AA-9.1#S0247">task_item</a></code>s after the reserved word <strong>private</strong> is called the private part of the task unit.] </p>
<p><strong>Proof: </strong>{"{"}<em>{"AI05-0299-1"}</em>{"}"} Private part is defined in Clause 8. </p>
<p>{"{"}<em>8652/0029</em>{"}"} {"{"}<em>{"AI95-00116-01"}</em>{"}"} For a task declaration without a <code><a href="./AA-9.1#S0246">task_definition</a></code>, a <code><a href="./AA-9.1#S0246">task_definition</a></code> without <code><a href="./AA-9.1#S0247">task_item</a></code>s is assumed.</p>
<p>{"{"}<em>{"AI95-00345-01"}</em>{"}"} {"{"}<em>{"AI95-00397-01"}</em>{"}"} {"{"}<em>{"AI95-00399-01"}</em>{"}"} {"{"}<em>{"AI95-00419-01"}</em>{"}"} {"{"}<em>{"AI05-0042-1"}</em>{"}"} For a task declaration with an <code><a href="./AA-3.9#S0078">interface_list</a></code>, the task type inherits user-defined primitive subprograms from each progenitor type (see 3.9.4), in the same way that a derived type inherits user-defined primitive subprograms from its progenitor types (see 3.4). If the first parameter of a primitive inherited subprogram is of the task type or an access parameter designating the task type, and there is an <code><a href="./AA-9.5#S0257">entry_declaration</a></code> for a single entry with the same identifier within the task declaration, whose profile is type conformant with the prefixed view profile of the inherited subprogram, the inherited subprogram is said to be <em>implemented</em> by the conforming task entry using an implicitly declared nonabstract subprogram which has the same profile as the inherited subprogram and which overrides it. </p>
<p><strong>Ramification: </strong>The inherited subprograms can only come from an interface given as part of the task declaration. </p>
<p><strong>Reason: </strong>{"{"}<em>{"AI05-0042-1"}</em>{"}"} The part about the implicitly declared subprogram is needed so that a subprogram implemented by an entry is considered to be overridden for the purpose of the other rules of the language. Without it, it would for instance be illegal for an abstract subprogram to be implemented by an entry, because the abstract subprogram would not be overridden. The Legality Rules below ensure that there is no conflict between the implicit overriding subprogram and a user-defined overriding subprogram. </p>

#### Legality Rules

<p>{"{"}<em>{"AI95-00345-01"}</em>{"}"} A task declaration requires a completion[, which shall be a <code><a href="./AA-9.1#S0248">task_body</a></code>,] and every <code><a href="./AA-9.1#S0248">task_body</a></code> shall be the completion of some task declaration. </p>
<p><strong>To be honest: </strong>{"{"}<em>{"AI05-0229-1"}</em>{"}"} If the implementation supports it, the task body can be imported (using aspect Import, see B.1), in which case no explicit <code><a href="./AA-9.1#S0248">task_body</a></code> is allowed. </p>
<p>{"{"}<em>{"AI95-00345-01"}</em>{"}"} {"{"}<em>{"AI95-00399-01"}</em>{"}"} [Each <em>interface_</em><code><a href="./AA-3.2#S0028">subtype_mark</a></code> of an <code><a href="./AA-3.9#S0078">interface_list</a></code> appearing within a task declaration shall denote a limited interface type that is not a protected interface.] </p>
<p><strong>Proof: </strong>3.9.4 requires that an <code><a href="./AA-3.9#S0078">interface_list</a></code> only name interface types, and limits the descendants of the various kinds of interface types. Only a limited, task, or synchronized interface can have a task type descendant. Nonlimited or protected interfaces are not allowed, as they offer operations that a task does not have. </p>
<p>{"{"}<em>{"AI95-00397-01"}</em>{"}"} {"{"}<em>{"AI05-0090-1"}</em>{"}"} The prefixed view profile of an explicitly declared primitive subprogram of a tagged task type shall not be type conformant with any entry of the task type, if the subprogram has the same defining name as the entry and the first parameter of the subprogram is of the task type or is an access parameter designating the task type. </p>
<p><strong>Reason: </strong>This prevents the existence of two operations with the same name and profile which could be called with a prefixed view. If the operation was inherited, this would be illegal by the following rules; this rule puts inherited and noninherited routines on the same footing. Note that this only applies to tagged task types (that is, those with an interface in their declaration); we do that as there is no problem with prefixed view calls of primitive operations for "normal" task types, and having this rule apply to all tasks would be incompatible with Ada 95. </p>
<p>{"{"}<em>{"AI95-00345-01"}</em>{"}"} {"{"}<em>{"AI95-00399-01"}</em>{"}"} For each primitive subprogram inherited by the type declared by a task declaration, at most one of the following shall apply:</p>
<p>{"{"}<em>{"AI95-00345-01"}</em>{"}"} the inherited subprogram is overridden with a primitive subprogram of the task type, in which case the overriding subprogram shall be subtype conformant with the inherited subprogram and not abstract; or</p>
<p>{"{"}<em>{"AI95-00345-01"}</em>{"}"} {"{"}<em>{"AI95-00397-01"}</em>{"}"} the inherited subprogram is implemented by a single entry of the task type; in which case its prefixed view profile shall be subtype conformant with that of the task entry. </p>
<p><strong>Ramification: </strong>An entry may implement two subprograms from the ancestors, one whose first parameter is of type <em>T</em> and one whose first parameter is of type <strong>access</strong> <em>T</em>. That doesn't cause implementation problems because "implemented by" (unlike "overridden') probably entails the creation of wrappers. </p>
<p>If neither applies, the inherited subprogram shall be a null procedure. In addition to the places where Legality Rules normally apply (see 12.3), these rules also apply in the private part of an instance of a generic unit. </p>
<p><strong>Reason: </strong>Each inherited subprogram can only have a single implementation (either from overriding a subprogram or implementing an entry), and must have an implementation unless the subprogram is a null procedure. </p>

#### Dynamic Semantics

<p>[ The elaboration of a task declaration elaborates the <code><a href="./AA-9.1#S0246">task_definition</a></code>. The elaboration of a <code>[single_task_declaration](./AA-9.1#S0245)</code> also creates an object of an (anonymous) task type.] </p>
<p><strong>Proof: </strong>This is redundant with the general rules for the elaboration of a <code><a href="./AA-3.2#S0024">full_type_declaration</a></code> and an <code><a href="./AA-3.3#S0032">object_declaration</a></code>. </p>
<p>[The elaboration of a <code><a href="./AA-9.1#S0246">task_definition</a></code> creates the task type and its first subtype;] it also includes the elaboration of the <code><a href="./AA-9.5#S0257">entry_declaration</a></code>s in the given order.</p>
<p>{"{"}<em>8652/0009</em>{"}"} {"{"}<em>{"AI95-00137-01"}</em>{"}"} As part of the initialization of a task object, any <code><a href="./AA-13.1#S0343">aspect_clause</a></code>s and any per-object constraints associated with <code>[entry_declaration](./AA-9.5#S0257)</code>s of the corresponding <code>[task_definition](./AA-9.1#S0246)</code> are elaborated in the given order. </p>
<p><strong>Reason: </strong>The only <code><a href="./AA-13.1#S0343">aspect_clause</a></code>s defined for task entries are ones that specify the Address of an entry, as part of defining an interrupt entry. These clearly need to be elaborated per-object, not per-type. Normally the address will be a function of a discriminant, if such an Address clause is in a task type rather than a single task declaration, though it could rely on a parameterless function that allocates sequential interrupt vectors.</p>
<p>We do not mention representation pragmas, since each pragma may have its own elaboration rules. </p>
<p>The elaboration of a <code><a href="./AA-9.1#S0248">task_body</a></code> has no effect other than to establish that tasks of the type can from then on be activated without failing the Elaboration_Check.</p>
<p>[The execution of a <code><a href="./AA-9.1#S0248">task_body</a></code> is invoked by the activation of a task of the corresponding type (see 9.2).]</p>
<p>The content of a task object of a given task type includes: </p>
<p>The values of the discriminants of the task object, if any;</p>
<p>An entry queue for each entry of the task object; </p>
<p><strong>Ramification: </strong>"For each entry" implies one queue for each single entry, plus one for each entry of each entry family. </p>
<p>A representation of the state of the associated task. </p>
<p>NOTE 1   {"{"}<em>{"AI95-00382-01"}</em>{"}"} Other than in an <code><a href="./AA-3.10#S0084">access_definition</a></code>, the name of a task unit within the declaration or body of the task unit denotes the current instance of the unit (see 8.6), rather than the first subtype of the corresponding task type (and thus the name cannot be used as a <code><a href="./AA-3.2#S0028">subtype_mark</a></code>). </p>
<p><strong>Discussion: </strong>{"{"}<em>{"AI95-00382-01"}</em>{"}"} It can be used as a <code><a href="./AA-3.2#S0028">subtype_mark</a></code> in an anonymous access type. In addition, it is possible to refer to some other subtype of the task type within its body, presuming such a subtype has been declared between the <code><a href="./AA-9.1#S0244">task_type_declaration</a></code> and the <code><a href="./AA-9.1#S0248">task_body</a></code>. </p>
<p>NOTE 2   The notation of a <code><a href="./AA-4.1#S0098">selected_component</a></code> can be used to denote a discriminant of a task (see 4.1.3). Within a task unit, the name of a discriminant of the task type denotes the corresponding discriminant of the current instance of the unit.</p>
<p>NOTE 3   {"{"}<em>{"AI95-00287-01"}</em>{"}"} {"{"}<em>{"AI12-0442-1"}</em>{"}"} A task type is a limited type (see 7.5), and hence precludes use of <code><a href="./AA-5.2#S0173">assignment_statement</a></code>s and predefined equality operators. If a programmer wants to write an application that stores and exchanges task identities, they can do so by defining an access type designating the corresponding task objects and by using access values for identification purposes. Assignment is available for such an access type as for any access type. Alternatively, if the implementation supports the Systems Programming Annex, the Identity attribute can be used for task identification (see C.7.1). </p>

#### Examples

<p><em>Examples of declarations of task types:</em> </p>
<CodeBlock>
task type Server is{"\n"}   entry Next_Work_Item(WI : in Work_Item);{"\n"}   entry Shut_Down;{"\n"}end Server;{"\n"}
</CodeBlock>

<CodeBlock>
{"{"}{"AI95-00433-01"}{"}"} task type Keyboard_Driver(ID : Keyboard_ID := New_ID) is{"\n"}      new Serial_Device with  -- see 3.9.4{"\n"}   entry Read (C : out Character);{"\n"}   entry Write(C : in  Character);{"\n"}end Keyboard_Driver;{"\n"}
</CodeBlock>

<p><em>Examples of declarations of single tasks:</em> </p>
<CodeBlock>
task Controller is{"\n"}   entry Request(Level)(D : Item);  --  a family of entries{"\n"}end Controller;{"\n"}
</CodeBlock>

<CodeBlock>
task Parser is{"\n"}   entry Next_Lexeme(L : in  Lexical_Element);{"\n"}   entry Next_Action(A : out Parser_Action);{"\n"}end;{"\n"}
</CodeBlock>

<CodeBlock>
task User;  --  has no entries{"\n"}
</CodeBlock>

<p><em>Examples of task objects:</em> </p>
<CodeBlock>
Agent    : Server;{"\n"}Teletype : Keyboard_Driver(TTY_ID);{"\n"}Pool     : array(1 .. 10) of Keyboard_Driver;{"\n"}
</CodeBlock>

<p><em>Example of access type designating task objects:</em> </p>
<CodeBlock>
type Keyboard is access Keyboard_Driver;{"\n"}Terminal : Keyboard := new Keyboard_Driver(Term_ID);{"\n"}
</CodeBlock>


#### Extensions to Ada 83

<p>The syntax rules for task declarations are modified to allow a <code><a href="./AA-3.7#S0061">known_discriminant_part</a></code>, and to allow a private part. They are also modified to allow <code><a href="./AA-9.5#S0257">entry_declaration</a></code>s and <code><a href="./AA-13.1#S0343">aspect_clause</a></code>s to be mixed. </p>

#### Wording Changes from Ada 83

<p>The syntax rules for tasks have been split up according to task types and single tasks. In particular: The syntax rules for <code>task_declaration</code> and <code>task_specification</code> are removed. The syntax rules for <code><a href="./AA-9.1#S0244">task_type_declaration</a></code>, <code><a href="./AA-9.1#S0245">single_task_declaration</a></code>, <code><a href="./AA-9.1#S0246">task_definition</a></code> and <code><a href="./AA-9.1#S0247">task_item</a></code> are new.</p>
<p>The syntax rule for <code><a href="./AA-9.1#S0248">task_body</a></code> now uses the nonterminal <code><a href="./AA-11.2#S0304">handled_sequence_of_statements</a></code>.</p>
<p>The <code><a href="./AA-3.11#S0086">declarative_part</a></code> of a <code><a href="./AA-9.1#S0248">task_body</a></code> is now required; that doesn't make any real difference, because a <code><a href="./AA-3.11#S0086">declarative_part</a></code> can be empty. </p>
